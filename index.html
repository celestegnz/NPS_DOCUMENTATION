Step #1 -> WAIT

En este paso, se define el periodo de espera para que las tablas necesarias estén actualizadas. Este paso es crucial para garantizar que los datos estén completamente cargados antes de continuar con el procesamiento.

Step #2 -> Query Segmentación

En esta sección se describe el proceso de segmentación de los datos utilizados para la muestra NPS. El código de segmentación permite organizar los datos según diferentes criterios, facilitando su posterior análisis. Este proceso ayuda a dividir los datos en grupos, lo que facilita el análisis de cada grupo de forma independiente y proporciona información más detallada.

Step #3 -> Muestreo

El código de muestreo es responsable de organizar y seleccionar un subconjunto de datos representativo, con el fin de obtener una muestra que refleje con precisión las características generales de un conjunto de datos más grande. Este proceso es clave para realizar análisis eficaces sin necesidad de procesar toda la información disponible, lo que podría ser costoso o innecesario.


query_param =f""" 
SELECT * FROM meli-bi-data.WHOWNER.LK_CX_NPS_PARAMETRIZACION WHERE UNIVERSE = '{universe}'
"""
    
universe_params = bq_gaia.query_to_df(query=query_param)
    
e_code_columns = universe_params['E_CODE_COLUMNS'].str.cat(sep=',').split(',')
concat_expression = "CONCAT(" + ", '_', ".join([f"COALESCE({col}, '')" for col in e_code_columns]) + ")"
    
query = f"""
    CREATE OR REPLACE TABLE {sample_table} AS (
    WITH 
    {concat_expression} AS PREV_ECODE,
    [...]
    SELECT
        E.SIT_SITE_ID,
        PREV_ECODE as E_CODE,
    [...] -> resto de columnas
    FROM STG_CX_NPS_PRECAP E
    LEFT JOIN meli-bi-data.WHOWNER.LK_CX_NPS_PARAMETRIZACION P
        ON E.PREV_ECODE = P.E_CODE 
        AND P.UNIVERSE_CAPS = '{universe}'
    LEFT JOIN HISTORICAL DW 
        ON E.CUS_CUST_ID = DW.CUS_CUST_ID 
        AND DW.SAMPLE_DATE < DATE_SUB(CURRENT_DATE(), INTERVAL CAST(P.SURVEY_FREQUENCY AS INT64) DAY)
    INNER JOIN meli-bi-data.WHOWNER.LK_KYC_VAULT_USER v on E.CUS_CUST_ID = v.cus_cust_id
        WHERE 
        RN<= CAP
    );
    """
bq_gaia.execute_query(query=query)

Paso a paso para el envío de encuestas

¿Necesitas entender cómo funciona el envío de encuestas?

Step #1 -> WAIT

Se espera al job de segmentación para poder avanzar.

Step #2 -> Survey Sent

En esta sección se crean dos tablas STG:

  • STG.MAX_SAMPLE_DATETIME
  • FINAL_TABLE

El propósito de estas consultas es asegurar que no se inserten registros duplicados en la tabla final (FINAL_TABLE) cuando se ejecute el trabajo múltiples veces. La lógica de las consultas es la siguiente:

  1. En la primera parte, se crea una tabla intermedia STG.MAX_SAMPLE_DATETIME que contiene el último SAMPLE_DATETIME para cada cliente (CUS_CUST_ID) dentro de un universo determinado (UNIVERSE). Solo se seleccionan aquellos registros que no han sido procesados anteriormente (es decir, los que no aparecen en STG.MAX_SAMPLE_DATETIME).
  2. En la segunda parte, se seleccionan registros de la tabla HISTORICA que coinciden con el último SAMPLE_DATETIME de cada cliente, asegurando que solo se inserten aquellos que no han sido procesados antes, gracias a la unión con STG.MAX_SAMPLE_DATETIME.

Step #3 -> SFTP TRANSFER

Este paso del proceso se encarga de recoger información sobre clientes y enviarla a un servicio llamado Qualtrics.

  1. Se establece una conexión segura a un servidor remoto donde se pueden guardar y enviar archivos. Este paso es como abrir una puerta que nos permite enviar nuestros datos a un lugar específico en internet.
  2. Se levantan variables por parámetro como:
    • stg_table: La tabla donde se encuentran los datos de cliente, stg_table = os.getenv('FINAL_TABLE').
    • csv_file: El nombre del archivo CSV, csv_file = os.getenv('CSV').
    • interruptor: Condición del interruptor, interruptor = os.getenv('INTERRUPTOR'). Si este interruptor tiene el valor "SI", se activará un condicional que enviará la encuesta solo a los usuarios de prueba (TEST) y obviará los resultados de la consulta. En caso contrario, si el interruptor tiene un valor de "NO", el envío será realizado a los usuarios seleccionados por la consulta, pero solo aquellos cuyo cust_id corresponda a la prueba (cust_id = os.getenv('CUST_ID')).
  3. Se prepara una consulta SQL, que es como una pregunta que hacemos a la base de datos. Esta consulta busca información sobre los clientes que queremos enviar, como nombres, correos electrónicos, y otra información importante.
  4. Ejecución de la consulta: Una vez que la consulta está lista, se envía a la base de datos para obtener los datos de los clientes. El resultado se guarda en un formato llamado DataFrame, que es una manera organizada de manejar datos en Python.
  5. Creación del archivo CSV: A partir de los datos obtenidos, se genera un archivo CSV (un tipo de archivo utilizado comúnmente para almacenar datos en forma de tabla). Este archivo contendrá toda la información que hemos recopilado, y su nombre incluirá la fecha de hoy para poder identificarlo fácilmente.
  6. Comprobación del archivo: Se verifica si el archivo CSV se creó correctamente. Si no se encuentra, se muestra un mensaje de error para asegurarse de que el proceso se está realizando como se espera.
  7. Envío del archivo al servidor remoto: Si el archivo existe, se transfiere al servidor SFTP donde se guardará. Este paso asegura que la información llegue al lugar correcto.
  8. Revisión de archivos en el servidor: Finalmente, se revisan los archivos almacenados en el servidor para confirmar que el archivo que se envió se encuentra allí, asegurando que el proceso ha sido exitoso.

Step #4 -> SAVE_HISTORICAL_CONSOLIDATED

Beneficios de la migración

La migración ha permitido una mejora significativa en la eficiencia operativa, optimizando recursos, reduciendo tiempos de procesamiento y aumentando la disponibilidad de servicios, lo que se traduce en un mejor rendimiento general y una mayor satisfacción de los usuarios.

  • Optimización de recursos

    • La migración permite hacer un uso más eficiente de los recursos, maximizando su rendimiento.

  • Reducción de tiempos de procesamiento

    • Se redujeron querys que pesaban 85Tb a querys que pesan 800GB.

  • Aumento en la disponibilidad de servicios

    • La migración puede aumentar la disponibilidad de servicios, haciendo que estén más accesibles para los usuarios y menos propensos a interrupciones o caídas del sistema.

  • Eliminación en encuestas duplicadas y usuarios TEST en tablas históricas

    • Evitamos el doble encuestamiento de usuarios y quitamos el sesgo de los usuarios test de las tablas de respuesta y muestreo histórico que ensuciaban la métrica.

Preguntas Frecuentes (FAQs)

¿Qué es la segmentación?

Consiste en dividir un conjunto de datos en grupos o categorías basados en criterios específicos. Su objetivo es permitir un análisis más enfocado y detallado de cada segmento.

¿Qué es el muestreo?

Proceso de seleccionar una muestra representativa de un conjunto de datos más grande para realizar análisis.

¿Por qué se realizó la migración?

Migrar datos de un universo de un sandbox a un datamesh implica mover los datos hacia una arquitectura descentralizada, donde los dominios responsables de los datos gestionan y optimizan su propio almacenamiento y acceso. El objetivo de esta migración es permitir que cada equipo o unidad dentro de la organización controle sus propios datos de manera más autónoma, al mismo tiempo que se mejora la escalabilidad y la eficiencia, y se facilita el acceso a los datos de forma segura y en tiempo real.

Un ticket de incidencia se utiliza para reportar problemas o fallos en el sistema, plataforma o procesos. Por ejemplo, si hay errores en un dashboard, un dato incorrecto, o cualquier inconveniente que afecte la operación.

Un ticket de solicitud se utiliza cuando se requieren cambios en elementos ya cerrados o configuraciones anteriores, como modificaciones de dashboards o mejoras que no son urgentes pero sí necesarias.
